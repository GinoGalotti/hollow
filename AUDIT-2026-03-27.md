## Part 3: Open Questions

### Q1: What's the current B6 balance target?
The ParameterizedBot (`smart` strategy) achieves 0% breach on standard+B2. The old `root_wide` shows 34% breach. The three-tier measurement model (root_wide / root_tall / smart) is now in place but hasn't been used to establish a new baseline. **What breach % target should we use, and measured against which strategy tier?**

### Q2: Should Provocation be base or pool?
The B6 redesign moved `presence_provocation` from pool to base. This was done to address the balance crisis (natives needed to be useful without stacking). But CLAUDE-balance.md §B6-v2 shows that with Provocation forced, breach drops from 34%→15% (root_wide). Is "always-on Provocation" the intended design, or should it return to pool once the optimizer finds good params?

### Q3: NetworkFear formula — D42 ≥3 neighbors vs old ≥2
D42 changed Network Fear from "≥2 adjacent presence territories" to "≥3 adjacent presence territories." On a standard 6-territory pyramid, only 2 territories (M1, M2) have 3+ neighbors. This means Network Fear only fires when an invader is at M1 or M2 AND their 3 neighbors all have presence. Is this intentional or too restrictive?

### Q4: Ember is completely untouched by B6
All the B6 work focused on Root. Ember's balance numbers (B1/B2/B4 era) haven't been re-validated against the D41 threshold redesign or the ParameterizedBot. The EmberBotStrategy still uses the old heuristic, not the new ParameterizedBot. Should Ember get the same ParameterizedBot treatment?
